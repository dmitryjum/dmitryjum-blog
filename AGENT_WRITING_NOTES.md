# Agent writing notes

These notes are for future agents writing or revising posts in this repo.

## Tone

- Write in a simple, direct style.
- Sound confident, but do not hype up the author or the work.
- Do not excuse decisions with too much framing like "this was old code" or "I would never do this now" unless that point is actually important.
- It is fine to sound interested in the work, but keep the tone steady.
- Prefer "here is what I built and why" over "here is why this was impressive."

## Style

- Keep sentences clear and readable.
- Prefer concrete explanation over dramatic setup.
- Avoid sounding promotional, self-congratulatory, or defensive.
- Do not overstate the cleverness of the code.
- Avoid turning every tradeoff into a grand lesson.
- Short paragraphs are good, but the post should still feel calm and natural.

## Content

- Focus on what the code does, why it was structured that way, and what problem it solved.
- Use more words when extra detail helps explain interesting code or behavior.
- If a section feels thin and there is meaningful code to unpack, expand it.
- If a section is mostly posture or self-commentary, cut it.
- Modernize the framing when useful, but do it lightly.

## Metadata

- Reuse existing tags when they fit.
- Do not use more than 4 tags per article.
- Include frontmatter in the same general shape as existing posts.
- Include a repo link at the end of each article when relevant.

## File naming

- Do not put dates in article filenames unless explicitly requested.

## Good defaults

- Be simple.
- Be clear.
- Be specific.
- Be confident without selling.
